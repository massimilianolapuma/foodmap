# Session 0007 — P11: Select the meal-planner model (#52)

- **Date:** 2026-05-31
- **Issue:** #52 (P11, area:data, type:feature)
- **Branch:** `feat/52-select-model`

## Goal

Let the user choose which engine backs meal-plan and suggestion generation,
behind the existing `MealPlannerAIService` protocol, privacy-first (on-device
options preferred) with a sensible per-device default.

## Changes

- `Domain/ValueObjects/MealPlannerModel.swift` (new)
  - `enum MealPlannerModel` — `automatic`, `ruleBased`, `onDevice` with
    localized `displayName` and `detail`.
- `Domain/Services/Services.swift`
  - New `MealPlannerModelStore` protocol (`selectedModel`, `select`,
    `isOnDeviceModelAvailable`) with a default `availableModels()` that hides
    the on-device option when the engine can't run.
- `Data/DataSources/UserDefaultsMealPlannerModelStore.swift` (new)
  - Persists the choice in `UserDefaults`; reports on-device availability via
    `SystemLanguageModel.default.availability`; degrades an unsupported
    on-device choice to `automatic`.
- `Data/DataSources/RoutingMealPlanner.swift` (new)
  - `MealPlannerAIService` that reads the preference fresh per call and routes
    to the rule-based or on-device planner (`automatic`/`onDevice` → on-device,
    which already falls back to rules).
- `App/AppContainer.swift`
  - Exposes `mealPlannerModelStore`; wires the routing planner (skipped under
    UI testing, which keeps the deterministic rule-based planner).
- `Features/Profile/ProfileView.swift`
  - New "Meal planner" section with an engine `Picker` (id
    `profile.plannerModel`) and a description line.
- Localized strings (en/it) for the new section.

## Tests

- `FoodMapTests/MealPlannerModelSelectionTests.swift` (new)
  - Default `automatic`, persistence, capability-aware on-device handling,
    `availableModels()` reflects capability, routing to rule-based vs on-device.
