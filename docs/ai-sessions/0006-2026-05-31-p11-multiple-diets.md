# Session 0006 — P11: Select multiple diet types (#50)

- **Date:** 2026-05-31
- **Issue:** #50 (P11, area:domain, type:feature)
- **Branch:** `feat/50-multiple-diets`

## Goal

Support selecting more than one diet type simultaneously, with combined meal
planning constraints and migration of existing single-diet data.

## Changes

- `Domain/Entities/UserProfile.swift`
  - Added stored `dietTypesRaw: [String] = []` (inline default → lightweight
    SwiftData migration backfills existing rows).
  - Added `dietTypes: [DietType]` computed accessor: getter falls back to the
    legacy `dietType` when no multi-diet selection is stored; setter normalizes.
  - Added `static func normalize(_:)` — dedups, preserves `DietType.allCases`
    order, collapses empty → `[.standard]`.
  - New init param `dietTypes: [DietType]? = nil` (keeps `dietType:` for compat).
- `Domain/UseCases/MealPlanningUseCases.swift`
  - `FilterMealsByDietRestrictionsUseCase` now requires a meal to satisfy
    **every** selected diet (`allSatisfy`), via a per-diet `satisfies(...)`
    predicate extracted from the old single-diet switch.
- `Data/DataSources/FoundationModelsMealPlanner.swift`
  - Prompt lists all selected diet display names joined by `, `.
- `Features/Profile/ProfileView.swift`
  - Replaced the single diet `Picker` with per-diet `Toggle`s bound to
    `dietTypes` (accessibility id `profile.diet.<rawValue>`).
